# 太阳系探索网站设计规格

日期：2026-06-17
状态：待评审草案

## 目标

建设一个高质量、交互式的太阳系科普网站。第一版要在视觉、美观度和交互顺滑度上达到较高水准，同时保持实现范围可控：通过模块化首页介绍太阳系相关主题，在需要的位置展示天体的近似相对位置，并通过引导式交互故事讲解典型天文现象。

产品主要服务于学生、教师和对天文感兴趣的普通用户。第一版优先保证清晰、好看、好用，不追求专业天文软件级别的完整性；同时保留后续接入更高精度星历数据和更深入模拟的架构空间。

## 非目标

- 第一版不追求 NASA 级别的轨道精度。
- 不做完整的天文馆系统或专业观测规划工具。
- 首屏不做营销式落地页。首屏应是一个视觉丰富的太阳系模块入口。
- 第一版不要求每个天文现象都做成完整物理模拟。优先选择重点模块做清晰讲解。

## 产品原则

1. 沉浸优先：用户一进入网站，就能看到一个有生命感的太阳系场景和明确的学习路径。
2. 探索和解释并重：每个交互控件都要和一个清晰的知识点关联。
3. 深度递进：普通用户可以快速浏览，感兴趣的用户可以继续深入查看。
4. 性能优先于过度真实：视觉效果不能牺牲交互流畅度。
5. 默认可访问：需要支持键盘操作、清晰对比度、减少动态效果模式，以及移动端友好的控件。

## 目标用户

- 正在学习基础天文学知识的初中和高中学生。
- 需要课堂可视化讲解工具的教师。
- 对太阳系运行、水星逆行、日食、月食、彗星等现象感兴趣的普通用户。

## 信息架构

网站包含六个主要区域：

1. **首页模块入口**
   - 首屏以太阳系作为背景：包括太阳、行星、轨道线和轻微动态。
   - 在空间场景上方或旁边展示清晰的学习模块入口。
   - 初始模块包括：
     - 太阳系概述
     - 日食
     - 月食
     - 水星逆行
     - 小行星带
     - 木星和它的卫星
   - 模块列表需要支持后续继续新增主题，而不必重做首页结构。

2. **太阳系仪视图**
   - 在“太阳系概述”模块中作为核心体验，也可在其他模块中作为辅助场景出现。
   - 根据当前模块展示太阳、八大行星、月球和相关小天体。
   - 支持时间控制、速度控制、标签、轨道线和相机预设。

3. **天体百科**
   - 为主要天体提供详情面板。
   - 包含物理参数、轨道参数、比例对比、显著特征和学习提示。
   - 可以从 3D 场景中点击天体打开，也可以从可搜索索引中打开。

4. **专题模块**
   - 为太阳系概述、日食、月食、水星逆行、小行星带、木星和它的卫星提供引导式讲解。
   - 每个模块结合滚动叙事、图解、互动节点和重点提示。
   - 第一版使用概念模拟和策划好的关键帧；后续版本可以接入真实星历窗口。
   - 每个模块拥有自己的局部场景。例如月食模块聚焦太阳、地球、月亮的相对位置，而不是展示完整太阳系。

5. **学习控制**
   - 时间滑杆和日期显示。
   - 速度选择：暂停、概念实时、按天推进、按月推进等。
   - 视图模式：真实比例提示、教学压缩比例、轨道俯视图、地心视角。
   - 图层开关：标签、轨迹、距离比例、轴倾角、月球轨道、事件标记。
   - 控件只在当前模块确实需要时出现；首页保持视觉清爽。

6. **教师和学生辅助功能**
   - 可分享的场景链接，包含当前模块、选中天体、日期、视图模式和故事步骤。
   - 术语表，解释轨道、黄道面、逆行、本影、半影、近日点、远日点等概念。
   - 轻量测验节点作为第二阶段功能。

## 第一版范围

第一版应包含：

- 一个带太阳系背景和六个模块入口的高质量首页。
- 一个位于“太阳系概述”模块中的 3D 太阳系场景，采用教学压缩距离。
- 可点击的太阳和行星信息面板。
- 由日期或时间滑杆驱动的近似轨道运动。
- 相机预设：完整太阳系、内行星、地球和月球、外行星。
- 六个引导式或半引导式模块：
  - 太阳系概述
  - 日食
  - 月食
  - 水星逆行
  - 小行星带
  - 木星和它的卫星
- 桌面端和平板端响应式布局。
- 移动端简化布局，减少控件和视觉密度。
- 减少动态效果模式：用静态图解和简单淡入淡出替代滚动驱动的 3D 编排。

## 后续范围

后续版本可以增加：

- 来自 NASA/JPL Horizons、SPICE/WebGeocalc 或 Skyfield 预计算结果的高精度星历数据。
- 历史和未来的日食、月食、水星逆行窗口搜索。
- 更多卫星、矮行星、柯伊伯带天体、彗星目录，以及更深入的小行星带数据。
- 新增专题模块，例如哈雷彗星、四季成因、潮汐、行星环、火星探索、航天器任务等。
- 课堂模式和课程播放列表。
- 互动测验和教师讲义。
- 多语言内容。

## 视觉设计

整体视觉应具有沉浸感和科学感，但不能变成难阅读的暗色泛科幻界面。

- 首页背景：深空黑色基底，太阳和行星以优雅的太阳系构图出现，配合轻微动态和轨道线。
- 主背景：深空黑色和细腻星点，避免大面积紫蓝渐变。
- 主要强调色：太阳金、轨道青、火星锈红、地球蓝绿、中性白灰文字。
- UI 表面：克制的半透明面板，保证强对比度，避免卡片套卡片。
- 字体：现代、清晰、易读的无衬线字体。只有真正的章节标题使用大字号；紧凑面板内使用更小、更密的信息层级。
- 图标：控件优先使用熟悉的线性图标，不要在图标更清楚的地方使用纯文字按钮。
- 卡片：只用于百科条目或故事节点等重复内容，圆角不超过 8px，除非后续设计系统另有明确要求。

## 交互模型

首页交互以模块选择为核心：

- 用户在太阳系背景上浏览不同主题模块。
- 鼠标悬停或键盘聚焦某个模块时，可以轻微高亮相关天体或空间区域。
- 点击模块后进入独立专题页面，进行聚焦讲解和互动。
- 模块布局需要足够紧凑，让用户看到多个学习方向，但不能变成营销式卡片墙。

专题模块交互以直接操作为核心：

- 拖动旋转相机。
- 滚轮或双指缩放。
- 点击或轻触天体以聚焦并打开详情。
- 拖动时间滑杆更新天体位置。
- 切换图层时不能造成布局跳动。
- 键盘快捷键只作为增强功能，不作为必须操作路径。

引导式故事：

- 滚动推进叙事步骤。
- 关键节点固定 3D 场景或图解，同时切换解释文本。
- 用户可以暂停、重播或跳转步骤。
- 每个故事都有一个简短的“观察重点”总结。

模块示例：

- 月食模块打开一个聚焦太阳、地球、月亮的场景。
- 第一步定义三个天体及其角色。
- 后续步骤展示三者对齐、地球本影和半影，以及地球上的观察者会看到什么。
- 除非需要上下文，否则该模块不展示所有行星。

## 动效设计

动效应服务于理解：

- 轨道运动需要连续、平滑；速度变化要足够渐进，避免用户失去方向感。
- 大幅相机移动使用 400-700ms 缓动。
- 面板打开和关闭控制在 200-300ms。
- 悬停和按压反馈应接近即时，通常小于 150ms。
- 滚动驱动的专题动画可以更具电影感，但单个转场不能显得拖沓。
- `prefers-reduced-motion` 必须关闭相机编排、视差和长时间滚动动画。

## 数据模型

第一版可以使用本地结构化数据集：

- 天体元数据：
  - id
  - 名称
  - 类型
  - 半径
  - 质量
  - 与太阳的平均距离
  - 公转周期
  - 自转周期
  - 轴倾角
  - 简短说明
  - 事实列表
  - 纹理或材质引用

- 轨道近似数据：
  - 半长轴
  - 离心率
  - 轨道倾角
  - 公转周期
  - 相位偏移
  - 展示比例

- 故事内容：
  - 故事 id
  - 标题
  - 摘要
  - 步骤
  - 相机目标
  - 日期或概念时间点
  - 标注
  - 图解叠层

- 模块元数据：
  - id
  - 标题
  - 简短描述
  - 分类
  - 首页视觉焦点
  - 路由
  - 难度等级
  - 预计阅读时间

第一版可以使用简化轨道公式或预计算位置样例。数据层必须隔离，方便后续替换为星历驱动的坐标，而不需要重写界面组件。

## 技术架构

推荐技术栈：

- Next.js + React 作为应用壳。只有当项目明确变成轻量单页原型时，才考虑 Vite。
- TypeScript 保证组件和数据结构的安全性。
- Three.js + React Three Fiber 负责 3D 渲染。
- Drei 提供常用 Three.js 辅助能力，例如控制器、加载器、标签和环境工具。
- GSAP ScrollTrigger 负责滚动驱动的专题章节动画。
- Motion 负责普通 UI 微交互。
- CSS Modules、Tailwind 或轻量设计 token CSS 层负责样式。
- Lucide React 负责界面图标。

主要模块：

1. **应用壳**
   - 路由、布局、响应式导航、全局主题和元数据。

2. **首页入口**
   - 太阳系背景构图、模块列表或网格、悬停和聚焦预览、响应式布局。

3. **太阳系仪引擎**
   - Three.js 场景、相机控制、光照、天体网格、标签、轨道线、点击拾取和渲染性能控制。

4. **时间系统**
   - 当前模拟日期、速度、暂停或播放状态，以及派生出的天体位置。

5. **天体数据**
   - 静态元数据和轨道参数。
   - 为 UI 和场景组件提供类型安全的数据访问。

6. **故事引擎**
   - 加载故事步骤，并映射到场景标注、相机状态和文字内容。

7. **UI 组件**
   - 面板、工具栏、滑杆、分段控件、开关、搜索、故事导航、术语表和响应式底部抽屉。

8. **状态持久化**
   - 将选中模块、选中天体、日期、视图模式和当前故事步骤写入 URL 状态。

## 性能要求

- 在现代笔记本和平板上保持流畅交互。
- 避免启动时加载所有高分辨率纹理。
- 尽可能使用压缩纹理。
- 在低性能设备上限制 3D 渲染的设备像素比。
- 延迟加载专题模块和重资源。
- 移动端优雅降级：减少标签、简化阴影、降低粒子密度。
- 动态标签或面板不能让核心控件发生意外尺寸变化。

## 可访问性要求

- 所有纯图标按钮必须有可访问名称。
- 所有控件必须可通过键盘访问。
- 焦点状态必须清晰可见。
- 普通文本对比度至少满足 WCAG AA。
- 减少动态效果模式必须真正可用，而不只是视觉上的象征处理。
- 只在 3D 中表达的信息，也必须能在文字面板中获得。
- 时间滑杆和开关状态必须能被屏幕阅读器理解。

## 内容要求

文案语气应准确、简洁、适合学生理解。不能为了简化而产生科学错误。

每个天体页面应包括：

- 一句话身份说明。
- 带单位的关键数字。
- 尺度对比。
- 轨道和自转说明。
- 三到五条易记事实。
- “为什么重要”的学习提示。

每个事件故事应包括：

- 这个现象是什么。
- 它为什么会发生。
- 哪种几何关系或时间条件造成了它。
- 地球上的观察者会看到什么。
- 常见误解纠正。
- 可选的可重播动画或图解。

## 错误处理

- 如果 WebGL 不可用，展示静态科普兜底页，包括天体索引和图解。
- 如果纹理加载失败，使用程序化材质兜底，并保持场景可用。
- 如果故事素材加载失败，展示文字步骤和静态图解。
- 如果 URL 状态无效，重置为默认视图，不能崩溃。

## 测试策略

初始验证包括：

- 轨道位置计算和日期时间状态的单元测试。
- 面板、控件、故事导航的组件测试。
- 桌面、平板和移动端视口的浏览器检查。
- 检查布局重叠和文字裁切。
- 在启用标签和轨道线时做性能冒烟测试。
- 减少动态效果模式测试。
- 如可行，测试 WebGL 兜底路径。

## 里程碑

1. **基础工程**
   - 项目搭建、设计 token、布局壳，以及带最小可用太阳系背景的首页模块入口。

2. **概述模块太阳系仪最小可用版本**
   - 太阳和行星、近似轨道运动、相机控制、标签、轨道线。

3. **天体百科**
   - 点击选择、详情面板、可搜索天体索引。

4. **时间控制**
   - 日期滑杆、播放或暂停、速度选择、视图模式。

5. **专题模块**
   - 太阳系概述、日食、月食、水星逆行、小行星带、木星和它的卫星。

6. **视觉打磨和可访问性**
   - 响应式布局、减少动态效果、键盘状态、视觉精修。

7. **验证**
   - 测试、截图、性能检查、问题修复。

## 待确认事项

1. 实现前需要确认行星纹理来源和授权。
2. 每个专题模块的正式讲解文案需要进行科普准确性审查。
3. 后续高精度版本使用哪个星历数据来源，应在第一版验证交互模型后再决定。

## 推荐方案

使用 Next.js、React、Three.js 和 React Three Fiber。第一版建设成一个中文优先的太阳系模块入口网站：以太阳系作为首页背景，在必要处使用近似轨道计算，采用真实感和教育化标注结合的视觉方式，移动端使用简化体验，并重点做好专题故事系统。数据层需要保持隔离，方便未来接入高精度星历数据和新增更多模块，而不重写界面。
