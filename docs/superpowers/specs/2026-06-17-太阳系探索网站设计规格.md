# 太阳系探索网站设计规格

日期：2026-06-17
状态：待评审草案

## 定位

建设一个可分享、可部署、可开源维护的高质量太阳系科普网站。第一版不是临时 demo，而是公开项目的第一个完整版本：功能范围保持克制，但视觉质量、工程结构、素材来源、文档和可访问性都要按正式网站标准处理。

第一版采用“沉浸首页 + 太阳系概述深做”的策略。首页展示六个学习主题，形成完整网站的方向感；其中只有“太阳系概述”可点击进入完整体验，其他五个专题只作为未来主题入口展示，不可点击，不提供预告页或专题路由。

## 目标

- 提供一个视觉高质量的太阳系入口网站，用户第一眼能感受到真实、沉浸、可信的空间场景。
- 完整实现“太阳系概述”模块，包含 3D 太阳系仪、近似轨道运动、时间控制、相机预设、天体百科和中英文切换。
- 使用真实公开影像或由真实观测资料制作的贴图表现太阳、行星和月球，不使用 AI 杜撰或无来源素材。
- 保持数据层、渲染层、UI 层和内容层清晰分离，方便后续扩展日食、月食、水星逆行、小行星带、木星卫星等专题。
- 让项目可以被他人拉取、安装、运行、构建、审阅和贡献。

## 非目标

- 第一版不做完整天文馆系统或专业观测规划工具。
- 第一版不追求 NASA/JPL 级别的星历精度。
- 第一版不实现日食、月食、水星逆行、小行星带、木星卫星五个专题页面。
- 第一版不做用户账号、收藏、课程播放列表、课堂模式或测验。
- 第一版不在运行时从互联网动态抓取图片或数据。

## 第一版范围

第一版只深做一个核心模块：**太阳系概述**。

必须包含：

- 首页 `/`：
  - 沉浸式太阳系背景。
  - 六个模块入口：太阳系概述、日食、月食、水星逆行、小行星带、木星和它的卫星。
  - 只有“太阳系概述”可点击。
  - 其他五个入口展示为“即将开放”或等价状态，不触发路由跳转。
  - 支持中英文切换。

- 太阳系概述 `/overview`：
  - 高质量 3D 太阳系场景。
  - 太阳、八大行星、月球、土星环、轨道线和星空背景。
  - 近似轨道运动和时间控制。
  - 播放/暂停、速度选择、日期或时间滑杆。
  - 相机预设：全景、内行星、地月、外行星。
  - 图层开关：标签、轨道线、月球轨道。
  - 点击天体聚焦并打开百科面板。
  - 中英文百科内容。
  - 桌面、平板和移动端响应式布局。
  - 减少动态效果模式。
  - WebGL 或纹理加载失败兜底。

## 后续范围

后续版本可以增加：

- 日食、月食、水星逆行、小行星带、木星和它的卫星等完整专题模块。
- 基于 NASA/JPL Horizons、SPICE/WebGeocalc 或 Skyfield 预计算结果的高精度星历数据。
- 历史和未来天文事件搜索。
- 更多卫星、矮行星、柯伊伯带天体、彗星目录和航天器任务。
- 课堂模式、课程播放列表、互动测验和教师讲义。

## 产品原则

1. 沉浸入口：用户进入首页即可看到有完成度的太阳系视觉场景。
2. 核心可用：第一版不铺开六个专题，而是把太阳系概述做到可反复使用。
3. 真实可信：3D 天体外观必须来自真实公开影像或真实观测数据派生素材。
4. 明确边界：未开放专题不能伪装成可用功能。
5. 开源友好：代码结构、素材来源、运行方式和授权说明必须清晰。
6. 默认可访问：支持键盘操作、清晰对比度、减少动态效果模式，以及移动端友好控件。

## 用户路径

1. 用户进入首页，看到动态太阳系背景和六个主题入口。
2. 用户点击唯一可用的“太阳系概述”。
3. 页面进入 `/overview`，展示完整太阳系视图。
4. 用户拖动或缩放 3D 场景，切换相机预设。
5. 用户点击地球、火星或其他天体，查看中英文百科面板。
6. 用户拖动时间滑杆或播放时间，观察行星沿近似轨道运动。
7. 用户可切换中英文，页面保持当前天体、视角和图层状态。

## 视觉设计

整体视觉应具有沉浸感和科学感，但不能变成难阅读的暗色泛科幻界面。

- 首页背景：深空黑色基底，太阳和行星以优雅的太阳系构图出现，配合轻微动态和轨道线。
- 主背景：深空黑色和细腻星点，避免大面积紫蓝渐变。
- 强调色：太阳金、轨道青、火星锈红、地球蓝绿、中性白灰文字。
- UI 表面：克制的半透明面板，保证强对比度，避免卡片套卡片。
- 字体：现代、清晰、易读的无衬线字体。只有真正的章节标题使用大字号；紧凑面板内使用更小、更密的信息层级。
- 图标：控件优先使用熟悉的线性图标，适合使用 Lucide React。
- 卡片：只用于模块入口或百科条目等重复内容，圆角不超过 8px。

## 3D 场景要求

3D 场景是第一版核心卖点，不能只是功能性示意图。

- 采用教学压缩比例，确保行星和轨道在屏幕上可读。
- 场景中必须明确提示“比例经过教学压缩”。
- 太阳要有发光材质、柔和光晕和主光源效果。
- 行星使用真实公开影像或由真实观测资料制作的纹理，不能杜撰外观。
- 地球应有云层或高光等额外细节。
- 土星需要可辨识的环。
- 星空背景需要细腻星点和空间深度，但不能喧宾夺主。
- 轨道线要克制、清晰、有层次，不能像调试线。
- 相机移动、聚焦天体和视角切换要顺滑。
- 桌面端优先保证观感；移动端可降低粒子密度、纹理分辨率和后处理强度。
- 如果某个纹理加载失败，使用程序化材质兜底，但需要保持视觉可接受。

第一版不做：

- 真实星历计算。
- 高精度椭圆轨道。
- 日食、月食、水星逆行等事件模拟。
- 航天器、矮行星、彗星和小行星带粒子系统。
- 复杂物理单位切换。

## 真实素材要求

太阳、行星和月球的 3D 视觉图像必须从可信公开来源获取，不使用 AI 杜撰图、随意手绘图或无来源素材。

优先来源：

- NASA Images and Media Usage Guidelines：https://www.nasa.gov/nasa-brand-center/images-and-media/
- NASA Photojournal：https://science.nasa.gov/photojournal/
- NASA Solar System Planets：https://science.nasa.gov/solar-system/planets/
- NASA/JPL/USGS 等机构发布的行星贴图、影像拼接图或任务照片。

素材处理规则：

- 每个素材必须记录来源 URL、机构、授权说明、下载日期和本地文件路径。
- 页面可以标注素材来源，但不能暗示 NASA、JPL、USGS 或其他机构背书。
- 如果页面标记了第三方版权，不能直接使用，除非另行取得授权。
- 如果没有适合的完整全球贴图，允许使用基于真实影像取样的派生贴图，但必须在素材记录中说明。
- 运行时不从互联网动态加载素材；素材应在开发期整理成本地资源。

## 多语言要求

第一版必须支持中文和英文两种界面语言，覆盖首页和“太阳系概述”完整体验。

范围包括：

- 首页六个模块入口的标题、说明和“即将开放”状态。
- `/overview` 的工具栏、相机预设、图层开关、时间控制和提示文案。
- 太阳、八大行星、月球的百科面板内容。
- 错误兜底、WebGL 不可用提示、纹理加载失败提示。
- README 中说明当前支持的语言。

交互要求：

- 页面提供清晰的语言切换控件，例如“中文 / English”。
- 默认语言可根据浏览器语言判断；无法判断时默认中文。
- 用户切换语言后应保持当前页面、选中天体、相机预设和图层状态。
- 语言选择需要持久化到 URL 查询参数，例如 `?lang=zh` 和 `?lang=en`，方便分享链接时保留语言状态。

数据要求：

- 面向用户的文案不能散落在组件里，应集中在 `i18n` 字典或本地化数据结构中。
- 天体百科数据需要支持 `zh` 和 `en` 两套文本字段。
- 英文内容需要自然准确，不能依赖未经审校的机器直译。
- 数字、单位和专有名词在两种语言中保持一致。

## 数据模型

第一版使用本地结构化数据，不依赖外部运行时 API。

### 天体数据

包含太阳、八大行星和月球：

- `id`
- 中文名和英文名
- 类型
- 半径
- 质量
- 平均日距
- 公转周期
- 自转周期
- 轴倾角
- 简介
- 三到五条事实
- “为什么重要”的学习提示
- 材质或纹理配置
- 素材来源引用

### 展示轨道数据

包含：

- 半长轴
- 展示距离
- 展示半径
- 公转周期
- 初始相位
- 轨道倾角
- 轨道颜色
- 是否默认显示标签

展示轨道数据只服务教学压缩太阳系仪，不声称是真实星历。

### 首页模块数据

包含六个主题：

- `id`
- 标题
- 简短说明
- 状态：`available` 或 `comingSoon`
- 首页视觉焦点
- 路由

只有“太阳系概述”状态为 `available` 并带 `/overview` 路由；其他五个状态为 `comingSoon`，不可点击。

### 素材元数据

包含：

- 天体 `id`
- 素材用途
- 来源标题
- 来源 URL
- 来源机构
- 授权或使用说明
- 下载日期
- 本地路径
- 是否经过裁切、压缩、拼接或贴图转换

## 技术架构

推荐技术栈：

- Next.js + React 作为应用壳。
- TypeScript 保证组件和数据结构安全。
- Three.js + React Three Fiber 负责 3D 渲染。
- Drei 提供相机控制、纹理加载、标签和环境辅助能力。
- Motion 负责普通 UI 微交互。
- Lucide React 负责界面图标。
- Tailwind CSS 配合轻量设计 token 负责样式，必要时为复杂 3D 覆盖层补充局部 CSS。

主要模块：

1. **应用壳**
   - 路由、布局、响应式导航、全局主题和元数据。

2. **首页入口**
   - 太阳系背景构图、模块入口、可用状态、不可点击的未来主题、语言切换。

3. **太阳系仪引擎**
   - Three.js 场景、相机控制、光照、天体网格、标签、轨道线、点击拾取和渲染性能控制。

4. **时间系统**
   - 当前模拟日期、速度、暂停或播放状态，以及派生出的天体位置。

5. **天体和轨道数据**
   - 静态元数据、轨道参数和类型安全的数据访问。

6. **素材系统**
   - 真实素材清单、来源记录、纹理路径和兜底材质配置。

7. **本地化系统**
   - `zh` 和 `en` 文案、百科内容、控件标签和错误提示。

8. **UI 组件**
   - 面板、工具栏、滑杆、分段控件、开关、语言切换、百科面板和响应式底部抽屉。

9. **URL 状态**
   - 保留语言、选中天体、相机预设和显示图层等轻量状态；语言使用 `?lang=zh/en`。

## 分层与扩展原则

网站需要为后续迭代保留扩展能力，但扩展不能影响已有功能。新增专题、更多天体、更高精度轨道或新学习工具时，应通过清晰接口接入现有系统，而不是改写核心页面和共享状态。

架构分层要求：

- **数据层**：负责天体数据、轨道参数、素材元数据和本地化内容，不直接依赖 React 组件或 Three.js 场景。
- **计算层**：负责时间推进、近似轨道位置、比例映射和派生状态，不包含 UI 展示逻辑。
- **渲染层**：负责 3D 场景、材质、光照、相机和拾取，只消费标准化数据和计算结果。
- **交互状态层**：负责选中天体、相机预设、图层开关、语言和 URL 状态，避免把状态散落到多个组件中。
- **UI 层**：负责首页入口、控制栏、百科面板、语言切换和错误兜底，不直接实现轨道计算或素材解析。

设计原则：

- 高内聚：每个模块只负责一个明确领域，内部逻辑集中，避免跨层散落。
- 低耦合：模块之间通过类型明确的数据结构和函数接口通信，避免组件互相读取内部实现。
- 单一职责：3D 场景不管理百科文案，百科面板不计算轨道，首页入口不承载专题实现。
- YAGNI：只为第一版和已明确的扩展点设计接口，不提前实现未确认功能。
- 回归保护：新增功能必须保留首页入口、太阳系概述、语言切换、素材来源记录和核心 3D 交互的既有行为。

## 开源质量要求

- 项目必须有清晰的 `README.md`，说明目标、功能范围、运行方式、构建方式、语言支持、素材来源和授权注意事项。
- 需要有素材来源文档，例如 `docs/assets.md`，记录所有真实影像和贴图来源。
- 代码结构要模块化，方便他人理解和贡献。
- 不把临时脚本、无来源素材、未压缩大文件或机器本地配置混入仓库。
- 生产构建必须通过，并支持部署到 Vercel 或同类平台。
- 页面不能暗示 NASA 或其他素材机构背书。
- 项目文档默认中文，技术名词、库名、API 名称、命令和路径可以保留英文。

## 开发原则

设计、开发和测试需要遵循 Karpathy Guidelines：

- 先明确假设、范围和成功标准，再开始实现。
- 优先选择能满足目标的简单方案，避免为未确认需求提前抽象。
- 修改保持克制，每一处改动都应能追溯到本规格或实现计划。
- 不做未要求的相邻重构；发现无关问题可以记录，不顺手改掉。
- 每个里程碑都要配套可执行验证，例如构建、测试、浏览器截图或素材来源检查。
- 如果实现中发现规格存在多种解释，先停下来明确取舍，不静默选择。
- 架构抽象以降低真实复杂度为目标，不为“未来可能会用到”而增加层级。

## 错误处理

- 如果 WebGL 不可用，展示静态科普兜底页，包括天体索引、说明文案和素材来源提示。
- 如果纹理加载失败，使用程序化材质兜底，并保持场景可用。
- 如果 URL 状态无效，重置为默认视图，不能崩溃。
- 如果语言参数无效，回退到中文。
- 所有错误提示需要支持中英文。

## 可访问性要求

- 所有纯图标按钮必须有可访问名称。
- 所有控件必须可通过键盘访问。
- 焦点状态必须清晰可见。
- 普通文本对比度至少满足 WCAG AA。
- 减少动态效果模式必须关闭长距离相机动画、视差和持续背景动效。
- 只在 3D 中表达的信息，也必须能在文字面板中获得。
- 时间滑杆、图层开关和语言切换状态必须能被屏幕阅读器理解。

## 性能要求

- 在现代笔记本和平板上保持流畅交互。
- 避免启动时加载过多高分辨率纹理。
- 尽可能使用压缩纹理或合理尺寸的本地图片。
- 在低性能设备上限制 3D 渲染的设备像素比。
- 移动端降低纹理分辨率、星点密度和后处理强度。
- 动态标签或面板不能让核心控件发生意外尺寸变化。

## 测试与验收

第一版验收按“可公开分享的网站”标准执行。

核心验收项：

- 首页在桌面、平板、移动端都能正常展示沉浸式太阳系入口。
- 首页只有“太阳系概述”可点击，其他五个专题不可点击并清楚标记“即将开放”。
- `/overview` 的 3D 场景稳定渲染：太阳、八大行星、月球、土星环、轨道线、星空背景都可见。
- 行星视觉来自有记录来源的真实公开影像或基于真实影像制作的贴图。
- 点击天体能打开对应百科面板，面板内容准确、可读，并支持中英文。
- 时间滑杆、播放/暂停、速度选择能驱动近似轨道运动。
- 相机预设能切换全景、内行星、地月、外行星。
- 标签、轨道线、月球轨道开关不会导致布局跳动。
- 中英文切换不重置 3D 场景状态。
- 中英文文本在不同视口下都不裁切、不溢出。
- 减少动态效果模式可用。
- WebGL 或纹理加载异常时有可接受兜底，不出现白屏。
- README 和素材来源文档完整，项目能被他人拉取、安装、运行和构建。
- 页面不能暗示 NASA 或其他素材机构背书。

验证方式：

- 单元测试：轨道位置计算、时间状态、模块可用状态、本地化字典完整性、素材元数据字段完整性。
- 组件测试：百科面板、首页模块入口、控制栏、图层开关、语言切换。
- 回归测试：后续新增功能时，必须覆盖首页可用入口、未开放入口不可点击、太阳系概述核心交互和中英文切换。
- 浏览器检查：桌面、平板、移动端截图，确认 3D 非空白、构图正确、文字不裁切、控件不重叠。
- 构建验证：`npm run build` 或等价命令必须通过。
- 可访问性冒烟：键盘可达、焦点可见、图标按钮有名称、文本对比度可读。
- 性能冒烟：首屏加载不能被过大纹理拖垮；移动端需要降低纹理、粒子和后处理强度。

## 里程碑

1. **基础工程**
   - 项目搭建、README、基础路由、设计 token、语言系统和部署配置。

2. **素材整理**
   - 查找太阳、八大行星和月球真实素材，整理本地资源和 `docs/assets.md`。

3. **首页入口**
   - 沉浸式太阳系背景、六个模块入口、唯一可点击的太阳系概述、五个不可点击未来主题、中英文切换。

4. **太阳系仪最小可用版本**
   - 太阳、八大行星、月球、轨道线、星空背景、相机控制和基础材质。

5. **高质量 3D 打磨**
   - 真实贴图、太阳光晕、土星环、地球细节、轨道线层次、移动端降级。

6. **学习工具层**
   - 时间控制、播放速度、相机预设、图层开关、点击聚焦和百科面板。

7. **多语言内容**
   - 首页、控制栏、百科、错误兜底和 README 的中英文内容。

8. **验证与修复**
   - 单元测试、组件测试、构建验证、浏览器截图、可访问性和性能冒烟。

## 风险与应对

- **真实贴图授权或分辨率不合适**：优先使用 NASA/JPL/USGS 来源，记录来源；必要时降低分辨率或使用真实影像派生贴图。
- **3D 质量和性能冲突**：桌面高质量，移动端降级；避免一开始堆叠过多后处理。
- **范围膨胀**：第一版只完成太阳系概述，其他专题入口不可点击。
- **科学表述不严谨**：百科文案保持短而准确，明确标注教学压缩比例和近似轨道。
- **中英文内容不一致**：本地化数据集中维护，测试缺失字段，避免组件内散落文案。

## 推荐方案

使用 Next.js、TypeScript、Three.js 和 React Three Fiber 建设一个中文优先、支持中英文切换的太阳系探索网站。第一版首页提供沉浸式模块入口，但只有“太阳系概述”可进入完整体验。3D 天体视觉必须来自真实公开影像或真实观测数据派生素材，并完整记录来源。项目按可公开分享、可部署、可开源维护的标准交付，而不是临时 demo。
