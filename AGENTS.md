# 项目约定

## 语言与文档

- 所有面向项目的文档、规格、计划、说明和评审记录默认使用中文。
- 技术名词、库名、API 名称、命令、代码标识符和文件路径可以保留原文。
- 面向用户的产品文案优先使用中文，除非后续明确加入多语言内容。

## 最高优先级：物理规律与天体运动规律

- **每次修改代码、模型参数、动画、阴影、比例压缩、相机视角、交互演示或测试时，都必须优先遵守物理规律和天体运动规律。**
- 当视觉表现、实现便利、交互体验与物理规律或天体运动规律冲突时，本节优先级高于其它实现规范；必须先保证物理机制正确，再处理视觉和交互体验。
- 任何教学压缩、视觉夸张或交互简化，都只能在明确不破坏核心天体关系的前提下进行；如果必须偏离真实尺度、速度或亮度，必须在实现和测试中保留正确的相对关系、运动方向、遮挡关系、光照方向和因果机制。
- 不能为了“看起来贴合”“动画顺滑”“实现方便”而改变太阳、行星、卫星、影锥、轨道、相位、遮挡、投影之间的真实几何关系；若视觉表现与物理机制冲突，先修正模型机制，再调整视觉表达。
- 涉及天体运动或天象演示的变更，动手前必须先说明对应自然规律、不可违背的边界和允许的教学抽象；验证时必须覆盖这些物理边界，测试命名和断言应直接体现物理约束，而不只是检查 UI 节点或静态截图存在。

## 工作方法

- 在方案设计、bug 修复、代码编写、问题复盘等环节，必须从第一性原理出发：先明确目标、约束、真实机制、失败原因和不可违背的边界，再选择实现方式；不要只套用既有模板或对表面现象做局部补丁。
- 任何需要进行代码、文档、配置、素材或测试变更的任务，在动手前必须先执行 `grill-me` skill，对目标、边界、风险、可选方案和不明确点进行追问；随后必须把所有拟改动点和疑问点与用户确认，确认后才能开始变更。
- 完成方案设计、bug 修复、代码编写等工作后，必须开启多Agent进行对抗式审查：主动寻找反例、边界条件、回归风险、与用户目标不一致之处，以及是否引入新的复杂度或视觉/行为副作用；审查结果要落实为修改或明确说明取舍。

## 天体模型与视觉素材

- 太阳、行星和月球的 3D 主模型优先使用真实公开影像、航天器观测资料或由真实观测资料制作的等距全球纹理，不使用简陋几何图形、曲线、色块或纯噪声来替代天体表面。
- 禁止使用粗糙的 patch、随机噪声、几个椭圆/色块、临时 canvas 涂抹或低保真程序化纹理来伪造任何天体表面。若需要表现月海、撞击坑、云带、风暴、环系统、日面活动等可见特征，必须基于真实公开影像、航天器观测资料、真实观测资料派生贴图，或质量足以匹配现有真实纹理的公开科学可视化素材；程序化叠加只能用于轻微 bump、受光、阴影、大气、遮挡或教学标注，不能替代天体本体纹理。
- 信息面板图片和 3D 表面纹理必须分离维护：
  - `purpose: "diffuse"` 用于信息面板全貌/代表图，保持可识别的天体全貌。
  - `purpose: "surface"` / `"clouds"` / `"ring"` 用于 3D 球体、云层或环系统贴图。
  - 不要为了 3D 细节把信息面板全貌图替换成局部细节图。
- 新增真实纹理时，必须同步更新 `src/data/assets.ts` 和 `docs/assets.md`，记录来源 URL、机构、授权/署名说明、下载日期、本地路径和处理方式。
- 3D 模型资源放在 `public/textures/`，文件命名使用 `body-purpose.ext`，例如 `moon-surface.jpg`、`saturn-rings.png`。
- 材质应服务天体真实性：
  - 岩质天体使用高 roughness、低 metalness，并可使用同一真实纹理作为 `bumpMap` 增强地貌。
  - 气态/冰巨星强调云层、大气和风暴纹理，不表现为固体陆地。
  - 暗弱行星环使用低透明度暗灰/深褐半透明材质，不使用高亮发光材质；必要时只加克制的受光侧高光和背光侧微弱散射。
- 对有显著特征的天体，需要用稳定命名的 mesh/group 表达特征，便于测试保护，例如大黑斑、土星环、天王星暗环、海王星 Adams 不连续环弧、月海/撞击坑纹理层等。

## 日食阴影几何

- 日食模块中的本影、半影和伪本影必须始终与太阳光线方向一致；阴影轴线必须沿太阳中心到月球中心的方向延伸，不能因贴合地表、截断体积或视觉避让而产生额外角度偏转。
- 半影圆锥体贴合地球表面时，只能沿同一太阳-月球光线方向求交或截断；不得把圆锥终点直接吸附到任意地表点后改变圆锥朝向。月球进入地球和离开地球的过程中，本影、半影、伪本影都必须保持同一套光线方向约束。
- 地表半影不是空间圆锥的纯圆形底面，而是太阳-月球光锥或光束与地球球面的交线、截痕或投影区域；它可能随入射角和地表曲率呈椭圆形、拉伸形或不规则闭合区域。空间半影体与地表半影痕迹必须分离建模和渲染，不能用移动或旋转圆锥底面的方式伪造贴地效果。
- 日全食的本影必须沿太阳-月球光线方向延伸并接触地球表面；不得为了避免穿模、视觉裁剪或让体积“看起来干净”而提前截断本影。地表全食带应来自本影与地球球面的相交区域。
- 日环食必须表达为本影锥尖在到达地球前结束，锥尖之后的伪本影沿同一太阳-月球轴线继续到达地表；地表环食带来自伪本影与地球球面的相交区域。不得把日环食画成本影直接打到地球表面的日全食机制。
- 修改日食阴影、切线、圆锥体、地表影带或点击热区时，必须从太阳圆面、月球圆面、月地距离和光线切线的几何关系推导；不得为了视觉贴合单独旋转或移动半影圆锥。
- 涉及日食阴影几何的测试必须覆盖日全食与日环食两种模型，并至少覆盖进入、食甚、离开三个时间点；测试应保护阴影轴线方向、圆锥终点、地表交点和影带痕迹，不只断言节点存在。

## 月食阴影几何

- 月食必须基于真实天体几何：太阳、地球、月球近似共线，顺序必须是太阳 -> 地球 -> 月球；地球位于太阳和月球之间，月球处于满月附近并穿过地球投向背日方向的影锥。
- 地球影锥轴线必须沿太阳中心到地球中心的反太阳方向延伸，不能指向太阳；影锥只能由太阳光被地球遮挡形成，不能为了贴合月球、构图或动画顺滑而移动、旋转或偏转影锥。
- 月球进入或离开影区必须由月球自身轨道运动造成；教学动画可以压缩距离、放大影锥或加快时间，但必须保留太阳光来自同一方向、地球在太阳和月球之间、阶段顺序和全食/偏食判定。
- 月食不是月球进入太阳的影子，也不是地球挡住观察者视线，而是地球挡住太阳直射到月球的光。月食只可能发生在满月附近；但不是每次满月都有月食，因为月球轨道平面相对地球绕太阳公转平面有约 5 度倾角，通常满月时月球会从地球影锥上方或下方掠过。只有满月发生在月球轨道交点附近，月球穿过地球半影或本影时，才会发生月食。
- 必须清楚区分三种影区：半影是太阳圆面只有一部分被地球遮挡的区域，月面只轻微变暗；本影是太阳圆面被地球完全遮挡的区域，月面明显变暗；月全食时整个月球进入本影，月偏食时只有部分月球进入本影。
- 月食本影和半影的空间轮廓必须由太阳圆面与地球圆面的切线生成，不能手写一套与太阳光线无关的圆锥半径。本影边界来自太阳与地球的同侧切线，例如太阳上缘 -> 地球上缘、太阳下缘 -> 地球下缘；半影外边界来自交叉切线，例如太阳下缘 -> 地球上缘、太阳上缘 -> 地球下缘。同一条同侧切线既是本影边界，也是半影内侧边界；半影外壁必须贴合交叉切线。教学压缩可以改变距离和半径数值，但太阳模型、地球模型、切线、影锥半径、可视太阳射线和月球影区判定必须复用同一套压缩后的几何输入。
- 月食太空视角中的太阳射线不是装饰线，必须复用生成本影/半影的同一套太阳-地球切线。不得额外绘制与影锥轮廓不重合的“光线”；若展示多条太阳光线，边界光线必须与半影外壁、本影外壁严格一致，其余光线只能作为内部辅助线且不能暗示新的影区边界。为了表达影锥是 3D 体积而不是单个平面截面，太空视角展示太阳切线/射线时应至少覆盖两个互相垂直的截面方向，例如 y 方向和 z 方向；两个方向必须由同一套太阳-地球半径与切线函数推导。
- 月全食必须包含并标注阶段：P1 月球开始进入地球半影、U1 月球开始进入地球本影、U2 整个月球进入本影、食甚、U3 月球开始离开本影、U4 月球完全离开本影、P4 月球完全离开半影。阶段顺序不能颠倒。
- 月偏食必须表现月球先进入半影，然后只有一部分月面进入本影，达到最大食分后离开本影，最后离开半影；月球圆面始终有一部分留在本影之外，不能出现整个月球都在本影中的阶段。
- 地球投射到月面的阴影边界必须呈圆弧状，体现地球影子的圆形截面；不得用水平裁切、矩形遮罩、任意黑色遮罩或不规则形状代替。
- 半影变暗必须明显弱于本影变暗。进入半影阶段只应轻微变暗，不能表现得像被黑色圆盘遮住；进入本影后，月面应出现清晰圆弧状暗影边界。
- 地球视角的月面必须以 NASA/SVS、NASA Images 或同等真实公开月球/月食影像为视觉基底，再叠加由月食几何驱动的半影、本影和红色折射过渡；不得用几个程序化椭圆、随机 crater、纯色 patch 或临时绘制纹理冒充真实月面。
- 地球视角月全食不能在进入本影时突然切换成整张血月图片；应随本影覆盖比例连续变化：半影先轻微变暗，本影覆盖区域先明显变黑，圆弧边界附近可出现暗红/铜红过渡带，整个月球完全进入本影后才呈现整体暗红但仍保留月面纹理。月偏食不应整体变红，被本影覆盖部分应显著变暗，未进本影部分只受半影轻微影响。
- 地球视角下的月食运动方向必须来自月球相对地球影区的视运动轨迹；当前月食地球视角要求月球水平穿过地影，因此地球本影/半影的圆形截面必须沿同一水平方向进入、覆盖并离开月面，不得误做成从上到下或从下到上的垂直扫动，也不得把地球影子画成矩形条、抛物线、任意 patch 或其它非圆形截面。月偏食也必须是圆形地影只覆盖月面一部分，而不是奇怪形状遮罩。
- 月全食阶段月球不能完全消失，应呈暗红、铜红或橙红色。红色来源必须解释为太阳光穿过地球大气层后发生折射和散射，短波蓝光被散射掉，较长波长的红橙光被折射进地球本影并照亮月面；不得表现或解释为月球自发光、地球发红光或太阳变红。
- 不要把月食画成地球圆盘直接覆盖天空中的月亮，不要让太阳和月球位于地球同侧，不要让月球在新月位置发生月食，不要让月偏食出现整个月球完全变暗的阶段，不要把半影表现得和本影一样黑。
- 涉及月食阴影、阶段、月面遮罩或红月表现的测试必须保护太阳-地球-月球顺序、影锥方向、月球轨道运动、P1/U1/U2/食甚/U3/U4/P4 阶段顺序、月全食整月入本影、月偏食永不整月入本影、圆弧状阴影边界、半影弱于本影和红月非自发光。

## 前端实现规范

- 保持现有 Next.js、React、Three.js、Tailwind 和 TypeScript 风格，不引入无必要的新框架或大型依赖。

- 每个非首页子页面都必须提供返回首页入口，统一使用共享的返回首页组件和样式；链接需保留当前语言参数。

- 修改浏览器 API 相关功能时，需要兼容 SSR/测试环境，先判断 `typeof window !== "undefined"` 和 API 是否存在。

### Next.js dev/build 缓存污染处理规范

- **问题表现**：访问首页时样式全部丢失，或终端出现 `Cannot find module './<chunk>.js'`，堆栈指向 `.next/server/webpack-runtime.js`、`.next/server/app/_not-found/page.js` 等 Next.js 构建产物。
- **根因**：已有 `npm run dev` / `next dev` 开发服务正在使用 `.next` 开发缓存时，又执行了 `npm run build` 或其它会重写 `.next` 的操作，导致开发服务引用的 server chunk、CSS 和静态资源与磁盘上的 `.next` 内容不一致。
- **标准解法**：停止当前项目的 Next dev 进程，清理 `.next`，再重新启动开发服务。
  - 推荐命令：`pkill -f '/Users/bytedance/projects/solar-system/node_modules/.bin/next dev' || true`
  - 如仍有包装进程：`pkill -f 'npm run dev' || true`
  - 清理缓存：`rm -rf .next`；如果目录为空但删除失败，可尝试 `rmdir .next`。
  - 重启：`npm run dev`。
- **预防要求**：运行 `npm run build` 前，先确认是否已有 dev server 正在运行；如果有，优先停止 dev server，或在 build 完成后主动重启 dev server。不要把 build 后的 `.next` 状态直接交给旧的 dev server 继续使用。
- **验证方式**：重启后访问首页，确认 HTML 中存在 `/_next/static/css/` 样式资源，并用 `curl -I http://localhost:3000/_next/static/css/...` 确认 CSS 返回 `200 OK` 与 `Content-Type: text/css`。

## Git 与文件编辑

- 工作区可能已有用户或其它任务的未提交改动；不要回滚、覆盖或格式化与当前任务无关的文件。
- 手工编辑文件使用 `apply_patch`，避免用 shell 重定向重写整文件。
- 新增或调整规范、素材、模型时，保持变更范围聚焦；不要顺手做无关重构。
